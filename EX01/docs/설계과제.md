# 설계과제 — 상품 관리 REST API (S-Product)

## 파트 A. 개발환경 구축 (NCS 1)
- `build.gradle` 에 web / data-jpa / validation / lombok / mysql / devtools 의존성을 구성한다.
- `application.properties`(공통) + `application-dev.properties`(개발 프로파일)로 DB 접속·포트를 분리한다.
- Git 저장소에 **역할별 커밋·머지·푸시**로 형상관리한다.
- 산출물: `docs/산출물_양식/개발환경_명세서.md`, `형상관리_방침.md` 를 작성한다.

## 파트 B. 공통모듈 구현 (NCS 2)
- 계층을 **Controller → Service(인터페이스+Impl) → Repository** 로 분리한다(결합도↓·응집도↑).
- 응답은 `ResponseEntity<Map<String,Object>>` 로 일관되게 래핑한다.
- 예외는 `@RestControllerAdvice` 전역 처리 + 비즈니스 예외 `MyBizException` 으로 분리한다.
- **JUnit 단위테스트**로 Service 의 정상/실패 동작을 검증하고, `테스트케이스_명세서.md` 에 조건을 명세한다.

## 파트 C. 서버 프로그램 구현 (NCS 3)
- `업무프로세스맵.md` 로 상품 업무 흐름(요청→검증→Service→Repository→DB)을 정리한다.
- REST CRUD 5종(목록/단건/등록/수정/삭제)을 구현한다.
- 등록·수정은 `@RequestBody @Valid ProductDTO` + `BindingResult` 로 검증하고, 오류 시 필드별 메시지를 담아 `400` 으로 응답한다.
- 등록·수정은 `@Transactional` 로 처리한다(Spring Boot 자동설정 트랜잭션매니저 사용).
- **보안(NCS 3.3)**:
  - ① 모든 입력은 `@Valid` 로 검증(음수 가격·재고 차단, 필수값 누락 차단)
  - ② JPA 파라미터 바인딩으로 SQL 인젝션을 차단(문자열 SQL 직접 조립 금지)
  - ③ 예외 메시지에 스택트레이스·내부 SQL 등 민감정보를 노출하지 않는다.

## 파트 D. 일괄처리 + 배치 프로그램 (NCS 4)
- **일괄처리**: `POST /api/products/bulk` — `List<ProductDTO>` 를 받아 **하나의 트랜잭션**으로 전부 등록한다.
  하나라도 검증/중복 위반 시 `MyBizException` 을 던져 **전체를 롤백**한다(부분 저장 금지).
- **배치(@Scheduled)**: `BatchConfig`(@EnableScheduling) + `ProductBatch.scheduleSoldOutCheck()` 로
  일정 주기마다 **품절(stock=0) 상품 수를 점검·로그**한다. 핵심 로직 `checkSoldOut()` 은
  `@Transactional(readOnly=true)` 로 DB 연동을 안전하게 수행한다(NCS 4.2).
- 배치 동작은 `ProductBatchTest` 로 검증한다(NCS 4.3).

## 제약
- 배운 범위(09~12, 04, 05) 밖 기술 사용 금지. 배치는 **Spring 기본 `@Scheduled`만** 사용(Spring Batch/외부 스케줄러 금지).
- 메서드 시그니처(이름·반환형) 임의 변경 금지. TODO 위치만 채운다.
- DataSource·EntityManagerFactory·TransactionManager 는 Spring Boot 자동설정을 사용하고, BatchConfig(`@EnableScheduling`)만 명시되어 있다.
